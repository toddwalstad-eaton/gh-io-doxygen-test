Detailed Description

Base "operation" class.

The bulk of the work to be done by the CDS is split up into abstracted "operations", which can be generated either directly from a user command (via libpx), or in response to some other operation.

Since the execution of one operation can trigger any number of additional operations, this forms the basic unit of work for the core CDS engine.

virtual void COperation::execute ( std::shared_ptr< px::cds::CRequestResponse >  msg,
COpList &  ops,
CDataStore &  cds

pure virtual

Execute the given operation.

This call implements an operation, performing some subclass-defined functionality.

Parameters
msgThe message which is currently being serviced and is therefore [indirectly] responsible for triggering this operation.
opsThe current operation list; while being executed, this object is at the front of the list. It may insert new operations into the queue as a result of execution.
cdsHandle to the main data store.
Exceptions
mcl_fetchThrown when the execution of this operation fails because a cached MCL entry is no longer valid. All operations must be recoverable from such an event, such that internal state remains consistent and they can be re-started afterward with no ill effects.
generic_errThrown when the execution of this operation fails for an unspecified reason. This condition is not fatal; the operation is simply removed from the queue, and the CDS continues.
try_againThrown when the execution of this operation needs to be preempted by some other operation. In this case, the operation is not removed from the queue; we simply fetch the highest-priority operation and begin execution again. This is useful for preempting an operation when it depends on another one before it can complete, e.g. a "set attribute" operation which triggers a change of interval, and therefore needs the interval window to be shifted before the actual "set" can complete.

const CThreadCommand* COperation::thr_cmd

Pointer to the thread command which is responsible for the execution of this operation.

This is NULL in the case of a normal operation, but is filled in with a pointer to the command (response, really) which contained it when it's a "chained" operation due to execution of a thread command. Most operations can ignore this member, but those that rely on returned thread data will need it.
