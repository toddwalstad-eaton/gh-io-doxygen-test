1 cdscmd Documentation {#mainpage}
2 ============
3 
4 
5  **Utility to provide command line access to the CDS APIs that are included in libpx.** <br>
6 This powerful tool is the "Swiss Army Knife" of command-line utilities for the Linux Toolkit.
7 It is useful for, or has been used in: <br>
8  - testing out CDS API functionality
9  - directly managing data in the CDS
10  - fetching and testing configuration by startup scripts
11  - providing CRUD operations for CDS data management in python regression tests
12  - browsing CDS contents
13  - showing specific updates in CDS data
14 
15 This program will exercise most of the CDS APIs (the exceptions being those whose parameters
16 are too complex for command-line entry).
17 
18 The **command line usage** illustrates the operation options for this program. <br>
19 First, the basic (relatively terse) usage information (obtained with the -? or -h options): <br>

357  -D :
358  --debug : Set the overall program debugging verbosity level
359 
360  This option sets the debugging verbosity level of the entire program.
361  Verbosity levels vary from 0, at which point only the most critical
362  messages are displayed, to >=10, beyond which every function call
363  prints its entry & exit point. Output is directed to stderr as well
364  as to the Linux system log daemon.
365 
366  (default: 1)
367 
368  -V :
369  -version : Print version information to stderr and exit
370 
371  This option prints usage information, including the Toolkit version,
372  build date, and revision number. However, be aware that version
373  information is only updated when this particular file is changed, so
374  this information is not necessarily up to date - if in doubt, you
375  should compare against a known firmware version.
376 
377  -ack :
378  -ackalarms : [<level> <alarm_id> [<alarm_id>...] [<user> [<notes>]] Ack these
379  alarm IDs at the given level. User and notes are included in each
380  ack if given.
381 
382  Acknowledge open alarms. Level is usually 1 or 2. User name and
383  note text are recorded in the alarm updates if given. Once a
384  latching alarm is acked at the same or higher level than any
385  trigger that has fired, when all trigger conditions have cleared,
386  the alarm is closed.
