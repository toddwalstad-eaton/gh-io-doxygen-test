cdscmd Documentation

Utility to provide command line access to the CDS APIs that are included in libpx.
This powerful tool is the "Swiss Army Knife" of command-line utilities for the Linux Toolkit. It is useful for, or has been used in:

  • testing out CDS API functionality
  • directly managing data in the CDS
  • fetching and testing configuration by startup scripts
  • providing CRUD operations for CDS data management in python regression tests
  • browsing CDS contents
  • showing specific updates in CDS data

This program will exercise most of the CDS APIs (the exceptions being those whose parameters are too complex for command-line entry).

The command line usage illustrates the operation options for this program.
First, the basic (relatively terse) usage information (obtained with the -? or -h options):

-D :
--debug : Set the overall program debugging verbosity level

          This option sets the debugging verbosity level of the entire program.
          Verbosity levels vary from 0, at which point only the most critical
          messages are displayed, to >=10, beyond which every function call
          prints its entry & exit point.  Output is directed to stderr as well
          as to the Linux system log daemon.

          (default: 1)

-V :
-version : Print version information to stderr and exit

           This option prints usage information, including the Toolkit version,
           build date, and revision number.  However, be aware that version
           information is only updated when this particular file is changed, so
           this information is not necessarily up to date - if in doubt, you
           should compare against a known firmware version.

-ack :
-ackalarms : [<level> <alarm_id> [<alarm_id>...] [<user> [<notes>]] Ack these
             alarm IDs at the given level. User and notes are included in each
             ack if given.

             Acknowledge open alarms. Level is usually 1 or 2. User name and
             note text are recorded in the alarm updates if given. Once a
             latching alarm is acked at the same or higher level than any
             trigger that has fired, when all trigger conditions have cleared,
             the alarm is closed.

-ada :
-ackdevicealarms : [<device id> [<device id>...] [<user> [<notes>]] Ack all
                   open alarms under the device recursively. User and notes are
                   included in each ack if given.

                   Acknowledge all open alarms under the selected device
                   recursively, regardless of the levels.User name and note
                   text are recorded in the alarm updates if given. Once a
                   latching alarm is acked and when all trigger conditions have
                   cleared, the alarm is closed.

-deletealarms : [-da [/ids <alarm_id>,[<alarm_id>...]] [/dids
                <device_id>,[<channel_id>]] [/r] [/et <end_time>] [/lvl <level>
                [/cnt <keep_count>]]]
                Delete all alarm sequences with given alarm ids. If alarm ids
                not given, delete the alarm sequences fulfilling the criteria
                given by other arguments

                If alarm ids are provided,
                Delete all closed alarm sequences with given alarm ids.
                If alarm ids are not provided, only delete alarms matching the
                following criteria:
                Alarm sequences which are closed before <end_time>.  Alarm
                sequences which never reached <level>.  Matching the device and
                channel ids in <device_id> (recursively if /r specified) or
                <channel_id>.  Of the alarms ids selected by above criterion,
                keep latest <keep_count> number of alarms.
                Examples are:
