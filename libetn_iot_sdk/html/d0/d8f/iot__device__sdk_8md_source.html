1 # Eaton IoT Device SDK
2 
3 ## Overview
4 
5 The Eaton IoT SDK is an API-based layer between your product code and the Azure "C" Device SDK. The Eaton IoT SDK has the following features:
6 
7 ### Manages the Connection to Azure {#md_connection_status_apis}
8 
9 * Your product code provides a configuration for the connection, and then uses the SDK to initiate the connection to an Azure IoT Hub.
10 * When Azure signals a change in the connection status (either good or bad), the IoT SDK will invoke a callback that your code provided so that your code can take the appropriate next action.
11 * The IoT SDK maintains an internal connection structure, and provides your code with a "handle" into the connection so your code can send messages.
12 
13 ### Provides the data structures compatible with PXWhite
14 
15 * The PXWhite services are a layer on top of the Azure cloud services that provide Eaton-specific data handling; think of Azure as the network provider and PXWhite as the database manager for the data sent.
16 * Azure is relatively agnostic about the content of the messages, but the IoT SDK structures the message content in the forms agreed to with PXWhite.
17 * The IoT SDK defines the structure for these Device-to-Cloud (D2C) messages:
18  * DeviceTree - message consisting of static information about each configured device, starting with the main or "Publisher" device.
19  * Realtimes - message containing the latest value and timestamp for one or more channels. PXWhite only holds the latest Realtimes values (ie, no history).
20  * Trends - message containing any combination of Last, Minimum, Average, or Maximum values for each channel in the message, with timestamp. This data is archived by PXWhite. Normally Trend messages are sent on a regular trend interval.
21  * D2C Commands - play a supporting role in firmware update from the cloud
22 * The IoT SDK defines the structure for Cloud-to-Device (C2D) messages:
23  * The Device Command - provides a command with key-value parameters
24  * Some PXWhite-defined commands exist, such as Get and SetChannelValues
25  * Same message structure can be used for product-defined messages
26  * The product code can build a response to return to PXWhite
27  * The Device Lifecyle Messages (DLM), used to do firmware update from the cloud
28 
29 ### Provides the services for message sending {#md_message_handling_apis}
30 
31 * Provides the function to send either
32  * A data message (DeviceTree, Realtimes, Trends)
33  * A D2C Command
34 * Provides internal serializing of the data for the message
35  * Serializes the data to a JSON string
36  * Any binary arrays are encoded as Base64 ASCII strings
37 * Provides a callback to signal when Azure confirms the message
38 * Starts a "messageThread" that runs periodically to dispatch any queued messages and lets Azure code "DoWork"
39 
40 ### Provides callbacks to the product code for cloud-driven events {#md_iot_sdk_callbacks}
41 
42 * The IoT SDK provides a method to register and unregister each supported callback
43 * The callbacks are:
44  * C2DDeviceCommandCallback for processing Cloud-To-Device device commands
45  * TimerCallback invoked periodically by the messageThread, if your product code needs to be woken up regularly
46  * ConnectionStatusCallback which is invoked when the status of the connection to IoT Hub changes.
47  * MessageDoneCallback which is invoked when a published Message (D2C) is confirmed by Azure (Success), or if it Fails.
48 
49 ### Provides Debug logging {#md_iot_sdk_logging}
50 
51 * Provides a means for the product to substitute its own debug logger for
52 the default logger in the IoT SDK.
53 * The default logging macros are defined in the product interface header (xxx_device_platform.h):
54  * Log_Error()
55  * Log_Info()
56  * Log_Debug()
57 * Lets the product set the maximum verbosity level of debug output.

67 ## Structure of the SDK
68 
69 The following directory tree shows the structure and purposes for the IoT Device SDK:
70 
71 ```
72  ├───docs - Documentation about the SDK
73  ├───include - Headers for the SDK source code
74  ├───make - Build instructions for the SDK as a library under linux
75  └───source - The "C" source code files for the SDK
76 ```
77 
78 ### Source code files
79 
80 The source folder contains the following source files:
81 
82  1. iot_device.c
83  - iot_open function – provided with ConnectionOptions, creates a messageThread to operate IOT things.
84  - iot_close function – closing down the existing connection and delete the Thread.
85  - iot_send function – Create message for the data to be pushed to the cloud.
86  - Callbacks registration for timer, c2dmessage, commands.
87  - List functions to create, delete lists, add and remove items from the list.
88 
89  2. iot_device_common.c
90  - Connection related functions – create connection, get connecttion details, delete connection, validate the connection options, initialize the connection.
91 
92  3. iot_serialization.c
93  - Serializing the Json data to be sent to the cloud from the device.
94  - Deserializing the Json format data coming from the cloud.
95 
96  4. iot_json_parser.c (JSMN - https://github.com/zserge/jsmn)
97  - Contains functions for parsing operations on json string.
98 
99  5. iot_encoding.c
100  - Contains functions to encode/decode Base64
101  - For safely communicating binary data
102 
103  6. azure_certs.c (file by Microsoft)
104  - Contains the "Baltimore" Certificate Authority file
105 
106 ### Header files
107 
108 The include folder contains the following header files:
109 
110  1. iot_device.h
111  - Contains constants, enums and structures for IT commands and requests.
112  - Contains callback declarations and function declarations.
113 
114  2. iot_json_parser.h (JSMN - https://github.com/zserge/jsmn)
115  - json parser header file
116 
117  3. pxred_device_platform.h
118  - The #defines to enable logging, enable protocols, for platform target PXRed.
119 
120  4. sample_iot_device_platform.h
121  - Example #defines to enable logging, enable protocols, and platform target (Px-Red or Px-Green).
