Detailed Description

Depdendence graph.

A dependence graph helps to track dependencies between source and target parameters. This is useful in any scenario in which the value(s) of one or more "sources" are used to derive the value(s) of one or more "targets", and more generally where targets may themselves be sources also. Two common examples are:

  1. Aggregation, in which any number of source channels may be combined to derive the value of a target channel, and that target channel may then itself be a source in some other derivation.
  2. Device / EDS equations, in which raw values are read into data "blocks", which then serve as sources for parameters (both internal, and external "channels"). The parameters themselves may then be chained together, i.e. these parameters are sources and targets.

A graph structure is a useful representation of these cases because it allows us to efficiently find things out about the relatively complex web of dependencies. In particular:

  • When adding a dependency, does it introduce a cycle in the graph? If so, evaluating it would result in an infinite loop, and the dependency should not be allowed. (For example, if a user attempts to add the aggregation equations: "A = B + C;" and "B = A + D;", we'd detect this as a bogus case.)
  • When updates occur at one or more source(s), which targets need to be re-evaluated, and in what order, to compute correct values even when the targets are chained together in a complex fashion?

This class hides the graph implementation used to answer these questions under the hood, and provides a simple interface that allows calling code to only concern itself with maintaining a list of sources and targets correctly.

void px::dep::CGraph::add ( const CNode &  node)

Add a target to the dependence graph.

The given target node is added as a vertex in our graph. Any sources that it depends upon are also added as vertices (if they don't already exist), and the dependencies between the target and its source(s) are added as graph edges.

Parameters
nodeThe target node to add to our graph
Exceptions
existsThrown if this target is already part of the graph; in this case, remove() should be called to remove the existing target first
bad_graphThrown if the addition of this target would create a cycle in the graph; in this case, the graph is left unmodified.

std::vector< px::dep::CNode::id_t > px::dep::CGraph::remove ( CNode::id_t  node)

Remove a node from the dependence graph.

This is called to remove a node when it is no longer required or supported. Any source nodes which are made obsolete as a result of this removal will silently disappear from internal tracking (although this is transparent to the caller). Any target nodes which used this node as a source will be returned as part of the "orphan list"; they are now potentially orphaned, and the caller must decide what to do with them.

Parameters
nodeThe ID of a node (source or target) to remove from our graph
Exceptions
unknownThrown if the identified node is not currently part of the dependence graph

std::vector< px::dep::CNode::id_t > px::dep::CGraph::evaluation_order ( const std::vector< CNode::id_t > &  nodes,
update_type  type = update_type::realtime 
) const

Get the correct evaluation order to satisfy all dependencies.

This call uses a topological sort to find an ordering of nodes which allows sequential evaluation that satisfies all dependencies correctly. In other words, if some targets depend on some sources, and those sources are in turn targets of other sources, etc. this call will figure out in what order we need to visit each target such that no targets are evaluated multiple times, yet all targets see up-to-date values for all of their sources.

This is only possible if our graph is a DAG; that's why the "add" calls fail if they would create a cycle in the graph.

The type of update is also accounted for by the dependence graph. Real-time updates are the usual case, but interval updates are handled by looking for interval-based dependencies between the sources and their immediate targets, and then looking for real-time dependencies between those targets and any others further down the line. If you don't know/care about the update type (for example, when dealing with Device Handler translation equations), just ignore this parameter and leave it at the default (real-time).

The result list includes the input nodes themselves, even if these nodes are not part of the dependence graph.

Parameters
nodesA list of source nodes whose values are to be updated
typeThe type of update which is about to occur for all IDs in the node list
