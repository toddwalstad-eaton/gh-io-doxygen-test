Detailed Description

Equation class.

This class supports the parsing and evaluation of equations. Equation syntax is C-like, and described in detail by an EBNF grammar utilized by Bison to generate the equation parser. Most equations which would work in a C program will also work with this class.

It has been noted that the order of precedence for operations is not entirely correct, eg, for power operations mixed with addition. However, this is presently seen as acceptable, as our users are probably not well enough versed in correct precedence anyway, so the solution is always to "when in doubt, use parentheses".

The functionality of this class is split into two phases, which operate quite differently:

  • The parsing phase takes a string equation as input, and builds an abstract syntax tree (AST) from the equation, much like a compiler. Scanning is performed by a Flex-generated token scanner, while parsing is performed by a Bison-generated LR parser. The scanning & parsing code is compiled in to this class, since it functions the same regardless of what kinds of objects are being translated. Parsing is relatively expensive, and should typically only be done once for a given equation.
  • The evaluation phase takes a "translator" as input – an object which is capable of getting and setting variables as they are encountered during equation execution. For example, in the equation "A = B + C;", the translator must 1) GET the value of "B", then 2) GET the value of "C", then 3) SET the value of "A". This code is template-based, which allows the caller to translate any kind of value they want, so long as the value conforms to the "Translatable" concept.

Common implementations of the "Translatable" object are provided by CSimpleVal and CValArray.

template<typename Translator >
void px::eq::CEquation::test_evaluate ( Translator &  t) const

This functions like evaluate(), except that it does a "test run" execution.

Test runs still call get() and set() on the given translator like a real execution, but all statements are evaluated rather than just those which are on some conditionally-true path.

For example, if given the equation: "if (a==0) b=c; else b=d;", test_evaluate() will always GET "a", "c", and "d". evaluate(), by contrast, will first GET "a", then depending on its value will either GET "c" or "d" (not both), depending on a's value.
