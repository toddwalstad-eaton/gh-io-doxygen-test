Detailed Description

Utility functions for handling permission values and sets thereof.

The overarching namespace for libpx.

Shortcut for toolkit config namespace.

All of the exported functionality for libpx is found under this px:: namespace.

This namespace is further divided by namespaces into the functional areas.

Many of the core toolkit objects, data types, and enums are defined at this namespace level because they are used across this library's functional areas.

typedef string px::shared_string

Currently we use ordinary strings for shared_strings as flyweight wrappers had subtle bugs.

At many places in Toolkit code, strings are useful for flexibility, but many copies of the same string might exist simultaneously. We use a flyweight wrapper in these cases to save memory.

Note that using a flyweight still requires us to construct a string object, so it doesn't save any time. But it does save space, so it's a good idea to use this anywhere where you're storing non-transient copies of strings, and those strings are likely to be duplicated in multiple places.

hashed_factory: Whether this or set_factory is better depends on the particular workload, but hashed is more scalable.

simple_locking: It's unknown whether these strings will be accessed from multiple threads at once, since that's a per-application decision. Therefore to be safe, we use synchronization by default.

refcounted: When all references to a flyweight string disappear, we'd like to free up the memory associated with it. To do that, we use reference counting to detect when each string is no longer necessary.

Conversion routines between protobuf objects and JSON strings.

JSON generated: one (or an array of) object(s) with the same name as the protobuf, with value of an array of objects, whose element names correspond to the names of the protobuf fields.

Examples: { "Channel":{ "canonical_name":"mACVAN", "display_order":53 } } or [ { "Channel":{ "id":123, "accumulated":false } }, { "Device":{ "id":317, "role":"publisher" } } ]

"repeated" fields become arrays, e.g.: "device_ids": [ 1, 2, 3 ] "message" fields become objects, e.g.: "minimum": { "float_val": 0.0 }

The Simplified or "Qualified" list form accepted by JSON2Protobuf(), but never generated by Protobuf2JSON(), useful when the list is all the same type of protobuf: Eg, here the protobuf_type might be "Channel" and the JSON string for 2 Channels: [ { "id":123, "accumulated":false }, { "id":456 } ]Convert a string containing JSON into an array of protobuf objects. Returns a string describing errors found, empty string if none. If any errors are found, parsing is probably incomplete. Elements whose name is not the same as an existing message field will cause a parsing error. Elements with a name corresponding to an existing message field, with a conflicting type (e.g. string value for an integer field) will return an error and not be parsed.

Parameters
