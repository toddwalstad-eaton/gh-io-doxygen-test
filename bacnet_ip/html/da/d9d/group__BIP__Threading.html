Threading design in bacnet_ip.
  • Very simply: Just one thread!
  • Boost ASIO manages the pseudo-multi-tasking of the various Actors
    • Everything is event-driven
    • Based on Boost::ASIO library (see the LibPX documentation)
    • Almost everything driven by “a response has come in on the serial port”
    • After initialization, BIP code never calls a blockable or time-indeterminate function (except in ASIO)
      • Because this could degrade its responsiveness when more than one client is addressing it.
    • The CDS Actor handles e.g. reading of channel values from CDS

The CDS API provides both synchronous ("wait for response) and asynchronous (with callback function) methods for data access (see the LibPX documentation):

  • OK to use the synchronous CDS methods during BIP initialization, when we are not waiting for serial input and so can afford to be blocked.
    • This results in simpler code.
  • Once polling has begun, must use
    • the asynchronous CDS methods with callback functions tied to the CDS Callback Actor.
    • or, in simple cases, the "fire and forget" synchronous methods
      • when we don't need to wait for the CDS response
      • eg, for pushing channel updates to the CDS. (In the unlikely case that one update is lost, will make up for it with the next update.)
