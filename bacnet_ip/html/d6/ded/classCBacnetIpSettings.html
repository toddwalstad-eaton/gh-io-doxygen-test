Detailed Description

BACnet/IP server settings (from Out of Band data).

Primarily looking at px::proto::BACnetSettings, but also others as well such as px::proto::BACnetUserSettings.

These settings are read from the CDS, and control how the BACnet/IP server operates. Settings are items mirrored in the CDS OOB objects. As such only items with which you would expect the web user or other process to control should be in here.

We use two separate maps for normal devices and decommissioned (or abnormal) devices. Decommissioned devices are those disabled or non-communicating. Normal devices go to _virtual_device_map. Decommissioned devices go to _virtual_decommissioned_device_map.

See Also
CBacnetIpConfig for an explanation of settings vs. configuration

Static function to use the the ethernet iMAC to get the default gateway 22-bit Object ID from the iMAC's lowest 22 bits, and to also create the MAC Template for the routed devices from it.

The ethernet MAC normally comes from the iMAC OOB object.

The MAC Template will start with the 3 lowest bytes of the iMAC (ie, the unique part after the vendor ID portion of the iMAC) and have 0 in the 4th byte and the default BACnet port 0xBAC0 in the last two bytes.

Parameters

buf_lenSize of both buffers; normally 6 bytes.
[out]mac_template_bufThe buffer to hold the new MAC template.
Returns
The computed Object ID (22-bits)

uint16_t CBacnetIpSettings::routed_network_number

Network number for the "virtual" BACnet network that the routed or "virtual" devices are "connected" to.

We cannot intuit a good starting value for the field, which must be unique among the enterprise's BACnet networks, so we must wait until this is configured.

As a special consequence, until this field is configured (non-zero), we will not start the BACnet services. This helps the case of those customers who do not care about BACnet and don't want their products "beaconing" I-Am message broadcasts at startup.

uint8_t CBacnetIpSettings::mac_template[BIP_MAC_LEN]

The BACnet "MAC" address template for the "virtual" devices, as a byte array.

For the routed devices, to fairly well ensure a unique MAC template, we will start with the 3 lowest bytes of the iMAC (ie, the unique part after the vendor ID portion of the iMAC) and have 0 in the 4th byte and the default BACnet port 0xBAC0 in the last two bytes.

Eg, if the iMAC is 00:20:85:F1:01:C3, then the MAC template will be F1:01:C3:00:BA:C0.

px::string CBacnetIpSettings::dcc_password

The (insecure) DeviceCommunicationControl password for all devices.

This password is required with DCC commands; if it has not been configured, then these commands will not be authorized.

Note that this password is sent "in the clear" in a DCC command, so it is inherently insecure. Maximum permitted length is 20 ASCII characters; we will add a requirement of minimum length of 6 characters. If the string set in the OOB is outside of this range, we will not copy it to this member.

bool CBacnetIpSettings::_only_shared_persisted_properties

Optional flag to force the persisted properties to be all of the "shared" variety, so all objects of the same object_type share any property listed in persisted_properties.

The default (false) is to have the properties of individual objects persisted, though some can be configured via OOB settings to be shared along with individual settings.

The primary impact of this flag is the action taken when a persisted property is set via BACnet; then if this flag is set True, the shared value will be set in persisted_properties for the given object_type. Otherwise, an individual object setting will be created or updated.
