Detailed Description

BACnet CDS Handler class (an ASIO Actor, specialized for bacnet_ip).

The communication with the CDS is handled by this class. Many of these methods use synchronous CDS calls (submit_wait()) for the programming flow convenience.

Every asynchronous callback, both solicited and unsolicited, from the CDS will invoke the handle_response() method, though many are actually handled elsewhere (eg, in CCdsHandler::internal_handle_response() for the internal device and channel management).

bool CBacnetCdsHandle::exec_device_cmd ( px::id_t  dev_id,
const px::string &  cmd_name,
const px::CValue &  value 
)

Utility function to execute a supported parameter-less device command.

The supported device commands for a given device were obtained by the list_device_commands() call, and each was mapped to a BACnet object, which is how the BACnet Object name happens to match a supported command name. Does a fire-and-forget when writing this out.

As currently implemented, using the px::cds::CSetFieldValue, with type set to px::cds::field_message_type::command, the value argument provided here is unused on the CDS receiving end. If a Value is a necessary part of the command, it must be added to the px::CDeviceCommand as the value part of a supported parameter and px::cds::CExecuteDeviceCommand must be used instead of CSetFieldValue.

Parameters
dev_idThe ID of the device that contains the channel.
cmd_nameThe command name.
valueThe Value to write to the CDS.
Returns
true if the cds request was successfully sent out, false otherwise.

hndlr_map_t CBacnetCdsHandle::_idle_hndlrs
protected

The map of all Polymorphic service handler objects, instantiated to execute the past BACnet function requests "owned" by this handler.

Each pending request will get one entry, keyed by its CDS transaction ID.

The handlers go here to wait for CDS responses. Handler entries should be removed once they are complete.
