Permissions for the "sysmond" user in System Monitor Operation

The files in this rootfs area are overlaid into the read-only file structure of the final target. Among the important functions here are the creation of the user named "sysmond" with specific permissions, as arranged in the file /etc/cds/uid_perms/13_sysmond.json and put into the system at build time.

Currently the permissions provided for the sysmond user allow it to do the things that the sysmond process needs, including the ability to do Device Commands with the control_basic and control_oper privileges.

The set of priviliges are enumerated in px::proto::Permission and similarly listed/described in CDS Session Roles in Confluence.

The product owner of the final product may choose to change these permissions, increasing or decreasing the list of privileges. To do so, the product must provide its own /etc/cds/uid_perms/13_sysmond.json file, which will be overlaid on the Toolkit sysmond's copy, replacing it.

The sysmond user for /etc/init.d/sysmond

The script that normally launches the sysmond process will put the name of sysmond in the command-line options for user and group when launching the process. The program will, after processing its command-line options, will change the user and group it is running under from root to the named user and group (here, "sysmond" for both).

The sysmond user is configured for the linux system by its presence in the following files: px-toolkit/rootfs/skeleton/etc/passwd and group.
