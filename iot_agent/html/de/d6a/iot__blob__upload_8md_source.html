1 # IOT Blob Upload {#iot_blob_upload}
2  Iot blob upload feature is used for uploading device data files as blobs to PX-White using Eaton and Azure SDKs.
3 
4 # Prerequisite for using IOT Blob upload:
5  1) Product needs to update "CONFIG_EATON_IOT_BLOB_ENABLE" to 'Y' for enabling feature. By default its set to 'N'.
6  2) IOT Agent should be Running.
7  3) IOTBlobUpload/filename OOB. This OOB can be created and deleted at product level.
8 
9 # Feature Info :
10 * Product team will initialize IOTBlobUpload/filename OOB with the file path, timestamp(optional) to be passed to PXWhite as blob timestamp, blob type and the CDS device id for which the file stores the channel/event data.
11 * IOT Agent will be subscribed to OOB and will be notified when created and updated.
12 * When subscription callback is received, the OOB will be checked for the request and response fields. If response is present, the OOB will be added to the pending list only if the result is INTERRUPTED and the retry count is less than the max allowed retries (set to 3 in PXRed). If no response is present, the OOB will be considered newly created and will be added to the pending uploads map.
13 * Every 5 secs(configurable by variable in IoT Agent device client), the pending list (map) will be checked and if pending uploads found, launch a thread to take care of the pending uploads.
14 * The thread will lock the pending list, copy the contents into a local list, clear the pending list and release the lock.
15 * For each item in the local list, IoT Agent will read the OOB and get the device id, file path, blob type and timestamps. Get the UUID from the device, open and read the file content into a buffer and pass these along with timestamp to the Eaton SDK blob upload function. These arguments would be concatenated to form the blobpath before passing to the EatonSDK.
16 * If the timestamp is not passed by the user, current timestamp will be used. And it will be written back to the OOB during setting the response to help the user to query the blob. Time can be specified upto miliseconds as "2020-01-01 12:12:12.123"
17 * Eaton SDK after processing the arguments, send the request to Azure SDK.
18 * Azure SDK responds with a success/fail once the blob upload is completed.
19 * Status of the upload is stored back into the OOB for each file and retry_count is updated.
20 * Thread will be completed after all OOBs in the local list are processed.

22 # How to test blob upload on BBB:
23 1. Log in to the BBB.
24 2. Start IOT Agent
25 3. Create a data file in file system under “/initlo/data” location. Add some text data in the file.
26 4. mkdir -p /initlo/data/blob ; vi /initlo/data/blob/testupload.txt
27 5. Create an OOB - " cdscmd -soj IotBlobUpload/uploadfile1 '{ "toolkitStruct":{ "iot_blob_upload":{ "blob_up_request": { "device_id":1, "filepath":"blob/testupload.txt", "blobtype":"firmware", "timestamp": { "sec": 1589448182, "nsec": 5 } } } } }' -pt pa -vv "
28 6. We can check whether OOB IotBlobUpload/uploadfile1 is created using the “cdscmd -got Iot” command.
29 6. Check Upload Status - "cdscmd -go IotBlobUpload/uploadfile1 -pp "

58 ## Use Swagger to check blob contents and list
59  To fetch the uploaded blobs through Swagger APIs, the device UUID, timestamp of the blob and blob type are required.User will need access to IOT hub swagger page.
60  * Log into Swagger, using the URL of your IoT Hub.
61 
62  We can use GET /api/v1/devices/{id}/blobs to fetch blob content for particular blob:
63 
64 1. In Blob tab click GET /api/v1/devices/{id}/blobs
65 2. Enter Device ID,Blobtype & time.and click "Try it out".
66 3. Ensure all time parameters are specified in correct format eg: '2015-09-01T14:30:56.123Z'
67 4. Gets the blob content from a device (identified by the device uuid), and based on the blob type and the blob timestamp.
68 
69 Alternatively we can use GET /api/v1/devices/{id}/blobs/list Api under Blobs tab :
70 
71 1. In Blob tab click GET /api/v1/devices/{id}/blobs/list
72 2. Time part under parameter is optional only date can be passed to fetch result.
73 3. Gets the list of blobs timestamps from a device, based on the blob_type and a time range. It will return no more than 5000 items. A device shall never push more than 5000 blobs per day. The 204 status code will be returned if the list is empty.
