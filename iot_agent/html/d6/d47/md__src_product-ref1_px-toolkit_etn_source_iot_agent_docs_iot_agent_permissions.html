Permissions for the "iot_agent" user in IoT Agent Operation

The files in this rootfs area are overlaid into the read-only file structure of the final target. Among the important functions here are the creation of the user named "iot_agent" with specific permissions, as arranged in the file .\04_iot_agent.json and put into the system at build time.

Currently the permissions provided for the iot_agent user allow it to do the things in system and CDS data that we anticipate the cloud-based user needing, as well as those the iot_agent process itself needs, including the ability to do Device Commands with the control_basic and control_oper privileges.

The set of priviliges are enumerated in px::proto::Permission and similarly listed/described in CDS Session Roles in Confluence.

The product owner of the final product may choose to change these permissions, increasing or decreasing the list of privileges. To do so, the product must provide its own \04_iot_agent.json file, which will be overlaid on the Toolkit iot_agent's copy, replacing it.

The run-time iot_agent role

The file \51_iot_agent_role.json creates a role named "iot_agent", which can be seen when browsing the roles in the UI, and can be edited - locally - if the user has security_admin privileges. (We do not envision the iot_agent user or role ever being given the security_admin privilege.)

However, as of the time of this writing, this role is not connected to the iot_agent user and so has no effect. Consequently, the build time choices for file 04_iot_agent.json will determine the priviliges for the iot_agent user.

The iot_agent user for /etc/init.d/iot_agent

The script that normally launches the iot_agent process will put the name of iot_agent in the command-line options for user and group when launching the process. The program will, after processing its command-line options, will change the user and group it is running under from root to the named user and group (here, "iot_agent" for both).

The iot_agent user is configured for the linux system by its presence in the following files: ......, group, and sudoers. It is necessary in the sudoers file to be able to shell out and run the firmware update and other system commands, without a password.
