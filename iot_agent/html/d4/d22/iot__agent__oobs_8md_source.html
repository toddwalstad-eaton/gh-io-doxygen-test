1 # Out-of-Band (OOB) Objects for IoT Agent Configuration and Operation {#md_iot_oobs}
2 
3 There are a variety of OOB objects that support the configuration and operation of the IoT Agent daemon (iot_agent):
4 
5 * IotConfigSettings
6 * H_IotConnectionString
7 * IotConnectionStatus
8 * ProxySettings
9 * IotDeviceMapping (unused)
10 * IotCatchup
11 
12 In addition, these OOBs for/from the Image Update program(s) are associated with the iot_agent's FirmwareUpdate
13 and Device Command operation:
14 
15 * ImageVersion/product/type/version OOBs
16 * ImageUpdate/xxxx
17 * ImageUpdateApp/product
18 * IotDeviceSettings
19 * IotBuiltInCommands/product
20 
21 This OOB handles iot_agent's Blob Upload operation :
22 * IotBlobUpload/filename
23 
24 These OOBs are each described in the sections below.

27 ## IotConfigSettings OOB
28 
29 The IotConfigSettings OOB provides the configuration information for the IoT Agent (iot_agent) process.
30 
31 The [default initialization for IotConfigSettings](../rootfs/etc/cds/generic/41_iot_agent_defaults.json)
32 does not enable the IoT Agent to run; if started, it will quickly recognize this and exit.
33 
34 The protobuf definition for the IotConfigSettings OOB is found in [IotAgentSettings.proto](../../libpx/tk_proto/hg/IotAgentSettings.proto).
35 
36 The sections below describe how the IotConfigSettings fields are used to support configuration
37 of the IoT Agent. Many of the fields are now deprecated; they were originally thought necessary, but have proved not to be.
38 
39 
40 ### Fields for the IotConfigSettings OOB
41 
42 The fields which are used in this OOB are listed individually.
43 
44 #### enabled
45 
46 When true, the IoT Agent is enabled to start and operate.
47 The factory default is false, and must be enabled by the owner,
48 if they want IoT access.
49 
50 #### hosted_uri
51 
52 Shows the URL for the Azure cloud address that the IoT Agent will attempt to connect to.
53 This field is filled in by the IoT Agent when it processes the connection_string,
54 and is for the convenience of the Net Diagnostics routines (which troubleshoot network issues).
55 
56 #### connection_string
57 
58 The owner must write to this field the Connection String for this device, which can be obtained
59 by giving PX White the UUID of this device (at the publisher level) and the adopter and site IDs.
60 
61 A valid Connection String consists of three parts: the hosted_uri, this device's (publisher) UUID,
62 and a Shared Access Token to authorize the connection.
63 
64 As soon as this field is written with an (apparently) valid ConnectionString, the IoT Agent code
65 will pull it out of this field, and substitute the text "**Securely Stored**" here. It will copy
66 the Connection String to the hidden OOB, H_IotConnectionString, which only the users with
67 Security Read or Write privileges can view. The reason for hiding the Connection String is
68 to protect the initial Shared Access Token provided in that string, to prevent spoofing a
69 connection from some other device.
70 
71 #### Unused (deprecated) fields
72 
73 The following fields are no longer used or of interest:
74 
75  * protocol
76  * use_json
77  * ping_time
78  * handshake_timeout
79  * not_tls
80 
81 ### Example of IotConfigSettings OOB
82 
83 The following shows an example for an enabled instance of this OOB:
84 
85 ```json
86 {
87  "toolkitStruct":
88  {
89  "iot_agent_settings":
90  {
91  "enabled":true,
92  "hosted_uri":"EatonTestIothub1.azure-devices.net",
93  "protocol":"IOT_TYPE_MQTT",
94  "use_json":true,
95  "ping_time":0,
96  "handshake_timeout":0,
97  "pong_timeout":0,
98  "not_tls":false,
99  "connection_string":"**Securely Stored**"
100  },
101  "description":"Configuration for the IoT Agent",
102  "label":"IoT Agent Settings"
103  }
104 }
105 ```

109 ## IotConnectionStatus OOB
110 
111 The IotConnectionStatus OOB is an OOB whose fields are set by the user (usually through the UI), and it provides
112 Security and User configuration (v1 and/or v3) input to the snmpd.conf file.
113 It also holds the trapsink and trap session information, which must be configured to be useful.
114 
115 This OOB is accessible by the UI but human access is restricted to Security Admin (for r/w)
116 and Security Viewer (for read-only).
117 
118 The [default initialization for IotConnectionStatus](../snmp_mibs/rootfs/etc/cds/generic/53_snmp_user_defaults.json)
119 disables both v1 and v3 access, but also provides the well-known v1 community strings of 'public' and 'private'.
120 Of course, no trapsinks are configured by default.
121 
122 The protobuf definition for the IotConnectionStatus OOB is found in [IotConnectionStatus.proto](../../libpx/tk_proto/hg/IotConnectionStatus.proto).
123 
124 
125 ### IotConnectionStatus OOB fields
126 
127 #### status
128 
129 The status of the connection to the Azure IoT Hub, from the set of IotConnectionStatusEnums.
130 It is IOT_OPEN on success.
131 
132 #### error_state
133 
134 A high-level error code for the IoT Agent's connection status, from the set of IotAgentErrorStateEnums.
135 Most useful when the status is Closed, as it then indicates the general reason why.
136 The error_reason text provides fuller details.
137 
138 #### error_reason
139 
140 The text here provides the Error Reason details: a human-readable reason for the error (if any).
141 
142 #### last_change_time
143 
144 The UTC time (in seconds) of the last change in status.
145 Equivalent to "time_t" in ANSI C, using the UTC timebase of seconds since the Epoch (Jan 1, 1970).
146 
147 #### traffic_count
148 
149 A measure of connection traffic, for diagnostic purposes.
150 The traffic_units indicates what is being counted.
151 
152 #### traffic_units
153 
154 The units or label for the measure of connection traffic.
155 As shown below, the default is to count the number of messages
156 which have been "confirmed" by Azure.
157 
158 #### log_changes
159 
160 The common OOB field is set to false here to prevent the
161 CDS from logging changes in this status OOB (since it changes
162 status frequently when the counts are updated, and is of no
163 special value to log the contents).
164 
165 
166 ### Example of IotConnectionStatus OOB
167 
168 The following example shows the status for a failed connection
169 (which is more interesting than for a successful connection).
170 
171 ```json
172 {
173  "toolkitStruct":
174  {
175  "iot_connection_status":
176  {
177  "status":"IOT_CLOSED",
178  "error_state":"IOT_DNS_FAILURE",
179  "error_reason":"DNS Service not responding",
180  "last_change_time":1534537222,
181  "traffic_count":0,
182  "traffic_units":"Messages Confirmed"
183  },
184  "description":"Status of the IoT Connection",
185  "log_changes":false
186  }
187 }
188 ```

191 ## H_IotConnectionString OOB
192 
193 The H_IotConnectionString OOB stores the IoT Connection String that was written to the
194 IotConfigSettings OOB's connection_string field. It is moved to this OOB, which is hidden
195 from most users, to hide the Shared Access Token that is part of that Connection String.
196 
197 This OOB would accessible by the UI (though there is little reason to do so),
198 but human access is restricted to Security Admin (for r/w) and Security Viewer (for read-only).
199 
200 The H_IotConnectionString OOB simply uses a single "GenericSettings" string field for its data, as defined in [ToolkitConfiguration.proto](../../libpx/proto/ToolkitConfiguration.proto), but then adds the read permissions to tightly restrict access.
201 The write and delete permissions are normal (not overridden here) since the security concern
202 is about revealing the SharedAccessToken to unauthorized parties.
203 
204 The sections below describe how the H_IotConnectionString fields are used.
205 
206 
207 ### Fields for the H_IotConnectionString OOB
208 
209 #### generic::StringData
210 
211 The IoT Connection String, given as one string of text.
212 
213 #### read_permission_required
214 
215 Declares the roles necessary for users (or programs) to be able to read this OOB.
216 Restricted to users with the Security Admin or View privileges.
217 
218 
219 ### Example of H_IotConnectionString OOB, using GenericSettings
220 
221 The following example shows this OOB with an (apparently) valid Connection String stored.
222 Note the read permissions, which make this a "hidden" OOB to most users.
223 
224 ```json
225 {
226  "toolkitStruct":
227  {
228  "generic":
229  {
230  "StringData": "HostName=EatonTestIothub1.azure-devices.net;
231  DeviceId=5731bd56-e445-54a1-a217-98e93fa14ba2;
232  SharedAccessKey=Ek9wHX2LX9g65tGGBmJxcpXWJy6qJf3VCZq5UeeLrDQ="
233  },
234  "read_permission_required": [
235  "SECURITY_READ",
236  "INTERNAL_APP",
237  "SECURITY_ADMIN"
238  ]
239  }
240 }
241 ```

244 ## ProxySettings OOB
245 
246 This OOB provides the proxy settings for outgoing connections. While in principle useful for
247 other connections, the only current use is for the IoT Agent's outgoing connection to the
248 Azure IoT Hub.
249 
250 By default, the proxy is disabled.
251 
252 The protobuf definition for the ProxySettings OOB is found in [ProxySettings.proto](../../libpx/tk_proto/hg/ProxySettings.proto).
253 
254 
255 ### Fields for the ProxySettings OOB
256 
257 #### proxy_enabled
258 
259 When true, the IoT must provide these proxy settings when attempting to open an
260 (outgoing) connection to the Azure IoT Hub.
261 
262 #### proxy_address
263 
264 Provides the Proxy server address or URI.
265 
266 #### proxy_login_name
267 
268 Provides the optional username for authenticating to the Proxy server.
269 
270 #### proxy_password
271 
272 Provides the optional password for authenticating to the Proxy server.
273 
274 #### internal_proxy_url (unused)
275 
276 This field is currently unused. In the past (ctc_agent), it provided
277 the path for the outgoing agent to use, which was input to the cntlm program;
278 cntlm then provided the proxying services.
279 
280 
281 ### Example 'ProxySettings'

283 Proxy settings for inside Eaton corporate networks, when enabled.
284 No username or password are required in this setting (fortunately).
285 
286 ```json
287 {
288  "toolkitStruct":

290  "proxy_settings":
291  {

293  "proxy_address":"https://proxy.etn.com:8080",
294  "proxy_login_name":"",
295  "proxy_password":"",
296  "internal_proxy_url":"http://127.0.0.1:3128/"

298  "description":"Proxy Configuration for the outgoing IoT Connection",
299  "label":"Proxy Settings"

313 ## IotCatchup OOB
314 
315 This OOB supports Trend Catch-up or "Publish after connection loss".
316 
317 It persistently records the time of the last successful trend interval publish, and any spans of
318 past time that remain to be published to catch-up PX White to the complete trend
319 records, so there will be no (or, at least, minimal) lost historic data.
320 
321 The protobuf definition for the IotCatchup OOB is found in [IotAgentSettings.proto](../../libpx/tk_proto/hg/IotAgentSettings.proto).
322 
323 
324 ### Fields for the IotCatchup OOB
325 
326 #### last_trend_published_time
327 
328 The timestamp of the last successfully published trend interval, in seconds since the epoch.
329 This will only be updated after all the trend messages for one interval from all devices have been
330 both published and ackowledged.
331 
332 #### trend_catchup_spans
333 
334 The list of trend catchup spans (if any) that should be caught up after successfully
335 connecting to the Azure IoT Hub. A span consists of a `"begin_span"` timestamp, and an `"end_span"` timestamp,
336 listing the first and the last times which must be published, inclusively.
337 
338 More than one span entry would mean that a previous catch-up process was interrupted
339 by another connection loss, resulting in additional gaps in the historic data.
340 
341 
342 ### Example 'IotCatchup'
343 
344 This example shows one catchup span entry in addition to the last published timestamp.
345 All times are in seconds since the Epoch of Jan 1, 1970.
346 
347 ```json
348 {
349  "toolkitStruct":
350  {
351  "iot_catch_up":
352  {
353  "last_trend_published_time":1576518000,
354  "trend_catchup_spans":[ { "begin_span":1576250100, "end_span":1576250100 } ]
355  },
356  "description":"Provides data about which records to publish for catch-up",
357  "label":"IoT Catch-up",
358  "log_changes":false,
359  "comments":"No trend_catchup_spans in the default."
360  }
361 }
362 ```

365 ## ImageVersion/product/type/version OOBs {#image_version_oob}
366 
367 The ImageVersion OOBs provide the version information for the indicated combination of product,
368 image type, and version to the IoT Agent (iot_agent) process. The "image type" is "LinuxImage" for
369 the main product PX Red firmware, but would be something else for other types of device firmware images
370 that have an image update app to support updating their firmware.
371 
372 There is no default initialization for ImageVersion OOBs; instead, these are created by their
373 image update application with its `make_version_oob` action, usually at startup.
374 
375 The protobuf definition for the ImageVersion OOBs is found in [ImageUpdate.proto](../../libpx/tk_proto/hg/ImageUpdate.proto).
376 
377 The sections below describe how the ImageVersion fields are used to describe the version information.
378 
379 ### Fields for the ImageVersion OOBs
380 
381 #### device_ids
382 
383 A list of one or more CDS Device IDs covered by this Image Version OOB.
384 All listed Devices must have the same version information; otherwise,
385 another ImageVersion with a different version in the name should be added.
386 
387 #### addl_capabilities
388 
389 The set of flags which indicate additional capabilities that the
390 target device is capable of (eg, the Reboot command),
391 besides Image Updating, via its Image Update application.
392 
393 This value is composed of one or more of the AdditionalCapabilitiesFlags values, OR'd together.
394 
395 #### image_update_app
396 
397 The name of the Image Update application to be invoked to manage
398 the various Image Update actions, and to provide the necessary OOB data.
399 
400 #### images
401 
402 A array of ImageEntry structures.
403 There may be multiple ImageEntries listed for this set of device_ids.
404 The following section shows the fields for the ImageEntry structure.
405 
406 #### Fields for the ImageEntry structure
407 
408 ##### image_name
409 
410 The name for this type of Image entry.
411 This will be compared to that in any firmware update package to determine suitability of the upgrade.
412 
413 ##### image_version
414 
415 The present Image version (eg, firmware revision).
416 The Image Update application for this Image will compare this version to that in
417 an image update, to determine whether to allow the update.
418 
419 ##### board_name
420 
421 Hardware name or type (eg, board name) for this Image.
422 May be helpful when determining compatibility with new Images.
423 
424 ##### board_version
425 
426 Hardware version (eg, board revision) for this Image.
427 May be helpful when determining compatibility with new Images.
428 
429 ##### update_constraints
430 
431 Flags for Upgrade Limitations / Constraints.
432 This value is composed of one or more of the ImageUpdateConstraintFlags values, OR'd together.
433 
434 ##### image_version_details
435 
436 Additional details for the present Image version, for human, not machine, consumption.
437 This might be a long version ID not suitable for machine processing,
438 designatin as "Beta" or "RCx" version, etc).
439 
440 ##### component_details
441 
442 Additional Version information for components in this Image, one per line.
443 For human, not machine, consumption, these entries list version information
444 for major components within the Image (eg, Toolkit, UI, or library versions).
445 Each component described should get a separate line.
446 
447 
448 ### Example 'ImageVersion/PX-Red-BBB/LinuxImage/RI-10-278'
449 
450 This example shows the ImageVersion OOB for a BBB's LinuxImage
451 at version RI-10-278; this product/image type/version pattern
452 will be used for all OOBs generated by the image update app.
453 
454 ```json
455 {
456  "toolkitStruct":
457  {
458  "image_version":
459  {
460  "device_ids": [
461  1
462  ],
463  "images": [
464  {
465  "image_name":"LinuxImage",
466  "image_version":"RI-10-278",
467  "board_name":"Architecture: arm Board: bbblack",
468  "board_version":"Cpu: ARMv7 Processor rev 2 (v7l) ",
469  "update_constraints":8,
470  "image_version_details":"RI-10-278-g916b993",
471  "component_details": [
472  "PX Red Toolkit: 3.18.6",
473  "Linux Kernel: 4.14.51",
474  "GCC Compiler: arm-cortexa8-linux-gnueabi-gcc (GCC) 4.9.4",
475  "Build Date: 08/22/18 16:06"
476  ]
477  }
478  ],
479  "addl_capabilities":1,
480  "image_update_app":"update_linux_image"
481  }
482  }
483 }
484 ```

487 ## ImageUpdate/xxxx OOBs
488 
489 The ImageUpdate/xxxx OOBs are non-persistent status OOBs which are actually produced by the
490 image update apps (eg, update_linux_image).
491 
492 These image update apps are usually invoked by the IoT Agent (iot_agent) process.
493 If the image update app is invoked by the IoT Agent,
494 that Agent will provide the full tag name for the OOB and then subscribe to updates in that OOB.
495 The firmware update app will generally update this OOB at each "step" (however it is that the app defines
496 a step), and the IoT Agent will then push this information as a status update to Azure.
497 
498 There is no default initialization for ImageVersion OOBs; instead, these are created by their
499 image update application with its `update` action.
500 
501 The protobuf definition for the ImageUpdate OOBs is found in [ImageUpdate.proto](../../libpx/tk_proto/hg/ImageUpdate.proto).
502 
503 The sections below describe how the ImageUpdate fields are used to indicate the status of an update process
504 for the IoT Agent.
505 
506 
507 ### Fields for the ImageUpdate/xxxx OOBs
508 
509 #### last_image_name
510 
511 The Image name or type currently being processed, or the last Image type acted upon when the process completes.
512 
513 #### last_device_id
514 
515 The CDS Device ID of the Device currently being updated, or of the last Device acted upon when the process completes.
516 
517 #### step_name
518 
519 A name for the step currently being carried out,
520 or the final step when the process completes through success or failure.
521 
522 #### step_number
523 
524 The number of the current step, within the expected series of steps.
525 
526 #### exp_total_steps
527 
528 The estimate of the total number of steps to complete the Image Update for all devices and all Image types.
529 
530 #### update_status
531 
532 The overall status of the Update, up to this point.
533 This will be one of the enums from the set of ImageUpdateStatusEnums.
534 
535 #### successes and failures
536 
537 Each is a list of the Image Update successes or failures, per Device + Image type.
538 Each entry is of the DeviceImageResult structure, which consist of the following fields.
539 
540 #### Fields for the DeviceImageResult entries
541 
542 ##### image_name
543 The name or type of the Image for this report.
544 
545 ##### device_id
546 
547 The Device ID of the Device for this report.
548 
549 ##### step_name
550 
551 The name for the (final) step at which the process failed. Not provided in success entries.
552 
553 ##### failure_reason
554 
555 Image Update Failure reason: Provides a human-readable reason for the failure of this
556 Device + Image type Update. Not provided in success entries.
557 
558 
559 ### Example 'ImageUpdate/xxxx'
560 
561 This example was captured at the (successful) end of a
562 FirmwareUpdate for the LinuxImage.
563 
564 ```json
565 {
566  "toolkitStruct":
567  {
568  "image_update_results":
569  {
570  "last_image_name":"LinuxImage",
571  "last_device_id":1,
572  "step_name":"script",
573  "update_status":"UPDATE_STATUS_DONE_SUCCESS",
574  "successes": [
575  {
576  "image_name":"LinuxImage",
577  "device_id":1
578  }
579  ],
580  "step_number":3,
581  "exp_total_steps":3
582  }
583  }
584 }
585 ```

588 ## ImageUpdateApp OOBs
589 
590 The ImageUpdateApp/product OOBs provide a mapping of the Image Update App names to
591 device models for use when a new device is added to the system and its Image Update App
592 must be called for the "make_version_oob" action to add this device to the correct
593 ImageVersion/product/type/version OOB.
594 
595 The default initialization for ImageUpdateApp OOBs are normally set at the product level,
596 since it is up to the product to map the device models to their supporting Image Update apps.
597 There may be more than one ImageUpdateApp OOB, as there may be a number of device models supported.
598 
599 The ImageUpdateApp OOBs use the GenericTableEntry type, as described in [GenericTableEntry.proto](../../libpx/tk_proto/hg/GenericTableEntry.proto).
600 
601 The sections below describe how the GenericTableEntry fields are used in the ImageUpdateApp OOBs.
602 
603 
604 ### Fields for the ImageUpdateApp OOBs, using GenericTableEntry
605 
606 #### label
607 
608 This must specify the device canonical model name.
609 
610 #### description
611 
612 Text to describe this entry.
613 
614 #### value
615 
616 This must be a string ("str_val") naming the Image Update app to be used for this model.
617 
618 #### read_permission_required
619 
620 Can be specified to limit the access to an OOB.
621 In this case, it is not because it is secure information, but because we don't want
622 the UI to show it.
623

651 ## IotDeviceSettings OOB
652 
653 The IotDeviceSettings OOB provides persistent IoT Device configuration information for the IoT Agent (iot_agent) process.
654 For example, this OOB records which devices have had the "special" channels for RealtimesDisabled or TrendsDisabled set.
655 
656 There is no default initialization for IotDeviceSettings OOB; instead, these are created by TBD.
657 
658 The protobuf definition for the IotDeviceSettings OOB is a repeated list (ie, array) of the GenericDeviceSettingsEntry found in [GenericTableEntry.proto](../../libpx/tk_proto/hg/GenericTableEntry.proto).
659 
660 The sections below describe how the IotDeviceSettings fields are used to support configuration
661 of the IoT Agent.
662 
663 
664 ### Fields for the IotDeviceSettings OOB, using GenericDeviceSettingsEntry
665 
666 This OOB consists of a repeated list of GenericDeviceSettingsEntries, one per device
667 with IoT settings that need to be persisted.
668 
669 The following fields are used in this structure:
670 
671 #### cds_id
672 
673 The CDS Device or Channel ID for this entry's data.
674 
675 #### settings
676 
677 The array of settings (GenericTableEntry structures) data (the label and value fields mostly,
678 with the optional description field).
679 The description field might be repurposed for "additional details" or path to the setting.
680 
681

695  "cds_id":1,
696  "settings": [
697  {
698  "label":"trendsDisabled",
699  "value":

701  "str_val":"true"

703  }

705  }
706  ]

711 ## IotBuiltInCommands OOBs
712 
713 The IotBuiltInCommands/product OOBs provide the usage and availability information for
714 IoT Generic Device Commands for the IoT Agent (iot_agent) process.
715 
716 The default initialization for IotBuiltInCommands OOBs are normally set at the product level,
717 since it is up to the product to determine which Built-In commands should be available.
718 There may be more than one IotBuiltInCommands OOB, as the supported commands may come from different
719 areas of the product.
720

792 ## IotBlobUpload OOB
793 There is no default initialization for IotBlobUpload OOB, these are created by the application dynamically as per need to upload a file.
794 
795 The application is responsible for creating a new OOB for each file to be uploaded. The OOB tag will have the format as IotBlobUpload/xxx where xxx will be the file name of file to be uploaded.
796 
797 The IoT agent will read the request from from and write the response to the OOB. The product will be responsible for deleting the OOB as required by them.
798 
799 IotBlobUpload OOB can be divided in two sections IotBlobUploadRequest and IotBlobUploadResponse.
800 
801 IotBlobUploadRequest contains device id, blob type, timestamp, relative file path for each of the files.
802 
803 IotBlobUploadResponse contains result enum, blob path, response message and retry count.
804 
805 OOB description in https://confluence-prod.tcc.etn.com/display/LNXTK/IoT+Blob+Upload+Design.
806 Sample OOB with request and response after upload:
807 ```json
808 {
809  "toolkitStruct":
810  {
811  "iot_blob_upload":
812  {
813  "blob_up_request":
814  {
815  "device_id":1,
816  "filepath":"blob/batterydata.csv",
817  "blobtype":"blobs"
818  },
819  "blob_up_response":
820  {
821  "result":"IOT_BLOB_UPLOAD_DONE_SUCCESS",
822  "resp_msg":"Successfully uploaded the file blob/batterydata.csv.",
823  "retry_count":1,
824  "blob_path":"1d1e8f9e-fd5a-5afc-a28b-aa41ac87d312/blobs/2020-07-14/14-07-45-312"
825  }
826  }
827  }
828 }
829 ```
