Debugging the iot_agent

Refer to file px-toolkit/documents/Test_and_Debug_with_KVM.md for some fuller details.

I found it best to use Remote Start GDB debugging. That is the process described here.

  1. To set it up:
    1. In your config.local, set CONFIG_DEVEL?=y before building (to get debuggable output)
    2. Set up the Remote Debugger configuration as described in Test_and_Debug_with_KVM.md,
  2. Do a normal build (make, make fsimage). Then if you want to include the debuggable version of your iot_agent, do this before uploading the firmware image, from the root product folder: touch output/exported/bin/iot_agent make -j8 rebuild_fsimage (this will get the unstripped iot_agent) scp output/images.tar admin@10.xx.xx.xx:/tmp (to copy it to your BBB board). Then on your BBB: cd /tmp sudo fw_update.sh images.tar (you might find you need to add the -f option sometimes) sudo reboot
  3. After it reboots, ssh into your BBB.
    1. To open port 2345 in the firewall (if /etc/init.d/product_init didn't already do this), Using your SSH session, enter sudo iptables -I INPUT -p tcp --dport 2345 -j ACCEPT The -I inserts this entry at the top of the list.
    2. Confirm with sudo iptables -L
  4. Start the remote gdbserver on your BBB, running a non-daemonized iot_agent with a reasonably high debug output: sudo /bin/gdbserver --remote-debug localhost:2345 iot_agent -u iot_agent -g iot_agent -w 0 -d -D 9 This will then wait for you to start your Eclipse debugger.
  5. In Eclipse, start your remote gdb session.

Running under valgrind

If you have valgrind enabled, you will get it in the build (firmware image). Stop the iot_agent and run a debug version of it like this (with some valgrind/memcheck options that I like to use): sudo valgrind --leak-check=full --malloc-fill=0xAB --free-fill=0 iot_agent -u iot_agent -g iot_agent -w 0 -d -D 9

If valgrind has complaints about strlen or strcmp or the like and wants a better ld.so, then touch output/exported/lib/ld-linux.so.3 and make -j rebuild_fsimage and scp it over and update.

Debugging in libcurl

I had to do this, and it was rather difficult to arrange, so I'm making notes here. Unfortunately, the config for curl somehow strips out the -ggdb option that we need. So do this:

  1. Build everything normally.
  2. Clean the curl folders (but don't mrproper): make -C px-toolkit/vendor_source/build/curl clean
  3. Now edit the Makefiles that have been created in px-toolkit/vendor_source/build/curl/curl-7.45.0/Makefile and lib/Makefile under that. In both, find the CFLAGS entry, and add the "-ggdb" option, eg CFLAGS = -Os -ggdb ...
  4. Rebuild curl, eg with FOCUS=curl; export FOCUS; make tk_focused_vend_source
  5. Rebuild the image and get the unstripped libcurl: make -j8 rebuild_fsimage
  6. SCP it over and update the firmware.
