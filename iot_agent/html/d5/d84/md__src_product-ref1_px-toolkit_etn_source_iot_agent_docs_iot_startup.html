IoT Agent Startup and Connection to Azure

The iot_agent goes through a number of steps to determine its configuration and arrange itself before it enters the run() loop of the ASIO amework. These steps are:

  • Create a new toolkit::CSysLog object to manage syslogging
  • Register the signal handlers
  • Parse the command-line options
    • Will exit if there are any errors, or if command line usage is requested.
  • Fork off to a daemon process unless prevented by the command-line options.
  • Create the CIotAgentCore (the application's main loop)
  • Create and attempt to open the CDS Handle
    • If the CDS is not yet available, wait and keep trying
  • Do the necessary configuration for the main loop(), CIotAgentCore::config():
    • initializes the debug logging subsystem
    • reads the configuration ("settings") from persistent Out-of-Band (OOB) data
  • Then start CIotAgentCore::main() and
    • Create the Manager for the ASIO framework
    • Create IO Actors for the Signal Handling and watchdog, and add them to ASIO
    • Create an Actor to manage the CDS and add it to ASIO
    • Start ASIO running

Once entered, iot_agent stays in CIotAgentCore::main() forever unless we get an exception or are signaled to exit

  • On exit,
    • Stop the main loop() (if still running)
    • Deregister the signal handlers
    • Return the exit code

Process for Connection to the Azure IoT Hub cloud {md_azure_connect}

The process in the iot_agent for connecting to Azure (starting in CIotDeviceClient::connect()) is as follows:

  • Fill out the iot_connection_options structure:
    • Choice of protocol (we currently only use AMQP Websockets for PX Red)
    • Copy of the Publisher Device's UUID, the Connection String, and the Proxy Settings, if enabled.
  • Set the IotConnectionStatus OOB's status to IOT_CONNECTING.
  • Call the IoT SDK's iot_open()
    • This calls the low-level Azure connection functions and tries to establish the connection.
    • If this attempt fails immediately (which is not common), it returns a failure code.
    • But if the attempt appears to succeed, that is not proof that the connection has really been established; that won't be known until the first message is acknowledged by Azure.
  • Assuming success,
    • Register the callbacks for Connection Status and Cloud-to-Device (C2D) Device Commands
  • Send the first messages:
    • DeviceTree for all known devices (active or inactive)
    • Device State of the Publisher (aka, "Device Realtimes") - ignored by PX White, but used as a sort of heartbeat in PX Red
    • For each communicating Device with valid Channels data, publish the real-time data for every channel once, here, the first time
      • The iot_send() function is called once for each valid Device's realtime channel data
    • For each communicating Device, publish the latest Trend Interval Values (if they are available, meaning if the CDS has been running long enough for one interval's worth of data)
      • The iot_send() function is called once for each valid Device's trend data
    • Check every time through the timer callback that drives the CIotDeviceClient state machine to see if we have received any message confirmations
      • When this first happens, change the IotConnectionStatus OOB's status to IOT_OPEN.
