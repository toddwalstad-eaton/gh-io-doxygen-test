CDS API

To programmatically access the Core Data Store, use the classes documented here.Construct a subclass of px::cds::CRequestResponse, then send it to the CDS process using px::cds::CClient::submit() or px::cds::CClient::submit_wait(). Either function will return the class to you, with the px::cds::CRequestResponse::status member set, and other data filled out as appropriate.

px::cds::CClient::submit_wait() will sleep until the response is received, then return a shared_ptr to the result. px::cds::CClient::submit() will return immediately, and the response will either appear in a queue managed by px::cds::CClient, or will asynchronously (on a background thread) call a px::cds::CCallback instance you have registered. It is also possible to ignore responses.

User, permissions, and session management

The CDS implements a role-based access control model - px::CUser users are assigned to roles, each of which gets a set of px::permissions. Client programs can also be assigned permissions, to implement a least-privilege model.

Subscription management

The CDS supports subscription mechanism for various Channels, Devices, OOBs etc These APIs are part of subscription and callback mechanism.

  • px::cds::CManageSubscriberIDs
  • px::cds::CSubscribe
  • px::cds::CSubscriberNotify
  • px::cds::CSubscribeSingle

Below are the steps mentioned to be followed for subscription.

step 1: Request Of New Subscriber ID.

In this step first it is decided which type of subscription should be done it can be OOBs subscription, Device Subscription Or Channel Subscription And respective subscriber id is requested. Api used to perform this is px::cds::CManageSubscriberIDs. This Api will return subscriber id size.

step 2: Submitting Subscription Request to CDS

This step further undergoes two process a) Adding of subscriber id and subscriber type. Api px::cds::CSubscribeSingle is populated by subscriber id and subscriber type (OOBs/Device). b) Final Submission Of Request to CDS. Api px::cds::CSubscribe has member px::cds::CSubscribeSingle as a vector. Object of CSubscribe is submitted to CDS.

Step 3: Registering Callback

px::cds::Client is used to register callback. This will call the class CCallback and register it. Generally cds_api_callback will handle the registeration response after some wait time. Response notification from cds will be handled by the class px::cds::CSubscriberNotify and notified to the subscriber or client.

In-Band (device, channel) data management

A channel is the unit of data in the Toolkit; channels have a value, and can have alarming and trending configured. Device objects serve to organize channels - they can contain channels and other devices, and they pass their states down the tree to contained devices and channels. Devices and channels are always stored persistently (except for state and channel value).

  • px::cds::CCreateDevice, px::cds::CCreateChannel

  • px::cds::CUpdateDevice, px::cds::CUpdateChannelSettings
  • px::cds::CDeleteDevice, px::cds::CDeleteChannel
  • px::cds::CGetChannelValue, px::cds::CUpdateChannelValue
  • px::cds::CSetDeviceState, px::cds::CSetChannelState

Out-of-Band (configuration) data management

For data that doesn't fit a device or channel model, one can use OOB (out-of-band) objects. These are instances of px::proto::toolkitStruct (see the protobuf definitions in libpx/tk_proto for details on the different possible data structures). OOB objects can be stored persistently or not. A string "tag" is associated with each object.

  • px::cds::CGetOOBTags, px::cds::CGetOOBData
  • px::cds::CSetOOBData, px::cds::CClearOOBData

Get/Set single fields of many different kinds of objects

Alarms

Any channel may have alarms configured. Some have alarms configured by default in their Master Channel List (MCL) entries. Alarms can be read, configured, and managed using these APIs, and can be subscribed to with px::cds::CSubscribe. Alarms do not trigger if the channel's state "disabled" or "disarmed" bits are on, and setting either of them closes any open alarms on that channel.

  • px::cds::CGetAlarmConfig, px::cds::CSetAlarmConfig
  • px::cds::CGetAlarmUpdateGlobalCount
  • px::cds::CGetActiveAlarmsIds
  • px::cds::CGetAlarms
  • px::cds::CAckAlarms

Channel Value Trend Storage

Trending is configured in a channel's px::CChannelSettings::flags member.

When a channel has trending enabled, its value and/or min/max/average will be saved over every 5-minute interval (00:00:00 - 00:05:00, 00:05:00 - 00:10:00, etc.) to persistent storage. Old trend data is aggregated to longer intervals (15 or 60 minutes) or deleted, per settings in the DatabaseSettings OOB object.

  • px::cds::CQueryTrends
  • px::cds::CExportTrend
  • px::cds::CDeleteTrends

Device Command/Control

The device command workflow: device handlers register commands using px::cds::CSetDeviceCommands (they are set non-persistently so this must be done at each bootup). Then the device handler subscribes to px::sub::subscription_type::device_command events on the devices it handles. Other clients can use px::cds::CGetDeviceCommands to get the list and px::cds::CExecuteDeviceCommand to execute a command. When that happens, the device handler gets notified via px::cds::CSubscriberNotify that the command should be executed. It does the needful in a device-specific manner, using px::cds::CSetDeviceCommandResult to notify of status and/or completion (and clients can poll this status with px::cds::CGetDeviceCommandResult).
