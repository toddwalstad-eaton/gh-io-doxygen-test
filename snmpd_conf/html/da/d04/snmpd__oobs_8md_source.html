1 # Out-of-Band (OOB) Objects for SNMP Configuration and Operation
2 
3 There are a variety of OOB objects that support the configuration and operation of the Net-SNMP daemon (snmpd):
4 
5 * SnmpDataSettings
6 * SnmpUserSettings
7 * SnmpV3NewPassphrases/[username]
8 * SnmpV3UsmUser/[username]
9 * SnmpMibs
10 * ACCESS_CTRL/SNMP
11 
12 These OOBs are each described in the sections below.
13 All but the SNMP_MIBS and Access Control OOBs are processed by the snmpd_conf program, where they serve as
14 input to the snmpd.conf file that the program generates; see its description for more details
15 on operation of this program.
16 
17 The reason for having a variety of OOBs rather than just one as input to snmpd_conf is due to
18 the various origins of the data for each; some are product-specific, and others are user-specified.
19 
20 
21 ## SnmpDataSettings OOB
22 
23 The SnmpDataSettings OOB is a fixed entry OOB provided by the product build, and provides product-specific
24 input to the snmpd.conf file.
25 
26 Each product should provide a file to override the [default initialization for SnmpDataSettings](../snmp_mibs/rootfs/etc/cds/generic/52_snmp_data_defaults.json).
27 
28 The protobuf definition for the SnmpDataSettings OOB is found in [SnmpDataSettings.proto](../libpx/tk_proto/hg/SnmpDataSettings.proto).
29 
30 The sections below describe how the SnmpDataSettings fields are used to support configuration
31 of the snmpd.conf file.
32 
33 
34 ### Fields for the SnmpDataSettings OOB
35 
36 Only one relevant field is used in this OOB.
37 
38 #### sys_object_id
39 
40 The sysObjectId OID for this product.
41 This should be a unique-to-this-product OID (as a text string), assigned in EATON-OIDS.txt.
42 Required.
43 
44 #### submodules
45 
46 Not used. (Replaced by the SnmpMibs OOB.)
47 
48 
49 ### Example of SnmpDataSettings OOB
50 
51 The following shows some representative entries in this OOB;
52 this example is from the defaults for this OOB.
53 
54 ```json
55 {
56  "toolkitStruct":
57  {
58  "snmp_data":
59  {
60  "sys_object_id":"1.3.6.1.4.1.534.2.12",
61  "submodules": [
62  "entityMIB"

64  },
65  "description":"SNMP Data settings for the Test product; OID = eatonPowerChainDevice.",
66  "label":"SNMP Data Settings"

70 
71 The corresponding entry in the snmpd.conf file looks like:
72 
73 ```

75 ```

78 ## SnmpUserSettings OOB
79 
80 The SnmpUserSettings OOB is an OOB whose fields are set by the user (usually through the UI), and it provides
81 Security and User configuration (v1 and/or v3) input to the snmpd.conf file.
82 It also holds the trapsink and trap session information, which must be configured to be useful.
83 
84 This OOB is accessible by the UI but human access is restricted to Security Admin (for r/w)
85 and Security Viewer (for read-only).
86 
87 The [default initialization for SnmpUserSettings](../snmp_mibs/rootfs/etc/cds/generic/53_snmp_user_defaults.json)
88 disables both v1 and v3 access, but also provides the well-known v1 community strings of 'public' and 'private'.
89 Of course, no trapsinks are configured by default.
90 
91 The protobuf definition for the SnmpUserSettings OOB is found in [SnmpUserSettings.proto](../libpx/tk_proto/hg/SnmpUserSettings.proto).
92 
93 The sections below describe how the SnmpUserSettings fields are used to support configuration
94 of the snmpd.conf file.
95 Note that unless either enable_v1 or enable_v3 is set, there will be effectively no SNMP access.
96 
97 
98 ### v1 User Fields for the SnmpUserSettings OOB
99 
100 #### enable_v1
101 
102 The enable flag for SNMP version 1.
103 If enabled, the rocommunity and rwcommunity strings should be set.
104 
105 #### ro_community
106 
107 Community string for v1 read-only access.
108 Defaults to the universally known value of "public".
109 Only a single community string entry will be provided for, since
110 the v1 support is effectively deprecated by security concerns.
111 
112 #### rw_community
113 
114 Community string for v1 read-write access.
115 Defaults to the universally known value of "private".
116 
117 Expect that this will contain just the community name,
118 but the contents of this string will be provided to snmpd.conf
119 without interpretation, so it is possible to give it the
120 fuller view restriction following the format of:
121 `community [default|hostname|network/bits [oid|-V view [context]]]`
122 
123 Only a single community string entry will be provided for, since
124 the v1 support is effectively deprecated by security concerns.
125 
126 ### v1 Trap Fields for the SnmpUserSettings OOB
127 
128 #### trap_community
129 Community string to be sent with trap messages.
130 Defaults to "public".
131 Can be used on the receiving end to filter trap messages.
132 
133 #### auth_trap_enable
134 
135 Enables the sending of authFail traps when an SNMP request
136 is received with invalid credentials.
137 Defaults to false (disabled).
138 
139 #### trap_sinks
140 
141 An array of trapsinks - hosts to whom traps are to be sent.
142 Normally just a network host address or hostname is provided,
143 but the full form is supported: `host [community] [port]`
144 
145 
146 ### v3 User Fields for the SnmpUserSettings OOB
147 
148 #### enable_v3
149 
150 The enable flag for SNMP version 3.
151 If enabled, the v3 user information should be configured.
152 
153 #### Array of v3_users
154 
155 An array of SNMPv3 configured users and their access levels.
156 Each entry consists of the following fields:
157 
158 ##### v3_users::username
159 
160 Name of the user. The choice of name is arbitrary to snmpd. Required.
161 
162 ##### v3_users::use_auth
163 
164 Whether authentication is required to accept queries
165 using this username. Default/null = true, which is also the highly recommended choice.
166 Optional but highly recommended.
167 
168 ##### v3_users::use_priv
169 
170 Whether privacy (message encryption) is required to accept queries using
171 this username. Default/null = false.
172 Ignored if use_auth is false. Optional.
173 
174 ##### v3_users::allow_write
175 
176 Whether to allow this user to do SETs. GETs and GETNEXTs are always allowed.
177 Default/null = false. Optional.
178 
179 ##### v3_users::has_auth_passphrase
180 
181 Whether an authentication passphrase has been set for this
182 user. Set by the system when a passphrase is set.
183 Default/null = false. Optional.
184 
185 ##### v3_users::has_priv_passphrase
186 
187 Whether a privacy passphrase has been set for this user.
188 Set by the system when a passphrase is set.
189 Default/null = false. Optional.
190 
191 
192 ### v3 Trap Field for the SnmpUserSettings OOB
193 
194 #### trap_sess (trap sessions)
195 
196 An array of trapsess - hosts to whom traps are to be sent.
197 The trapsess directive in snmpd_conf configures the v3 trap type that
198 trapsink does not support.
199 The full form for entries that is supported are:
200 * `-v 1 [-c community] host[:port]`
201 * `-v 3 -u v3user host[:port]`
202 
203 
204 ### Example of SnmpUserSettings OOB
205 
206 The following example shows a fully configured OOB, with v1 and v3 enabled,
207 two v3 users configured (one for read/write access), and both a trapsink and
208 trap session configured.
209 
210 ```json
211 {
212  "toolkitStruct":
213  {
214  "snmp_user":
215  {
216  "enable_v1":true,
217  "enable_v3":true,
218  "ro_community":"public",
219  "rw_community":"private",
220  "trap_community":"public",
221  "auth_trap_enable":false,
222  "trap_sinks": [
223  "10.222.42.78"
224  ],
225  "v3_users": [
226  {
227  "username":"myRoUser",
228  "use_auth":true,
229  "use_priv":false,
230  "has_auth_passphrase":false,
231  "has_priv_passphrase":false
232  },
233  {
234  "username":"myRwUser",
235  "use_auth":true,
236  "use_priv":false,
237  "allow_write":true,
238  "has_auth_passphrase":false,
239  "has_priv_passphrase":false
240  }
241  ],
242  "trap_sess": [
243  "-v 3 -u myRoUser -l authNoPriv 10.222.42.78"
244  ]
245  }
246  }
247 }
248 ```
249 
250 The corresponding entries in the snmpd.conf file, with their helpful comments removed for brevity's sake, look like:
251 
252 ```
253 rouser 'myRoUser' auth
254 rwuser 'myRwUser' auth
255 
256 rocommunity 'public'
257 rwcommunity 'private'
258 
259 trapcommunity public
260 
261 trapsink 10.222.42.78
262 
263 trapsess -v 3 -u myRoUser -l authNoPriv 10.222.42.78
264 
265 authtrapenable 2
266 ```

268 ## SnmpV3NewPassphrases OOB
269 
270 Each SnmpV3NewPassphrases/[username] OOB is a transient OOB which is created by the user (usually through the UI)
271 when a new v3 user is added, and it holds the passphrase(s) for v3 authentication and encryption.
272 It is read by, and then deleted by, the snmpd_conf program, to reduce any chance of passphrases leaking;
273 snmpd_conf will subsequently create a SnmpV3UsmUser/[username] OOB to store the hashed versions of
274 these passphrase(s).
275 
276 This OOB is written by the UI but only by someone with the role of Security Admin.
277 
278 The protobuf definition for the SnmpV3NewPassphrases OOB is found in [SnmpV3UserPassphrases.proto](../libpx/tk_proto/hg/SnmpV3UserPassphrases.proto).
279 
280 The sections below describe how the SnmpV3NewPassphrases fields.
281 
282 
283 ### Fields for the SnmpV3NewPassphrases OOB
284 
285 The data in the fields here are unencrypted and
286 intended to exist only transiently, to reduce any chance of passphrases leaking.
287 
288 #### username
289 
290 The username for the SNMPv3 user.
291 
292 #### authentication_passphrase
293 
294 The authentication passphrase for the SNMPv3 user.
295 
296 #### encryption_passphrase
297 
298 The encryption ("privacy") passphrase for the SNMPv3 user. If left blank,
299 the authentication_passphrase is used as the encryption_passphrase.
300 
301 
302 ### Example of SnmpV3NewPassphrases OOB
303 
304 The following example shows an OOB for new user "myRwUser", with
305 passphrases configured for authentication and encryption.
306 (I think this is correct; couldn't capture a real example!)
307 
308 ```json
309 {
310  "toolkitStruct":
311  {
312  "snmp_v3_userpass":
313  {
314  "username":"myRwUser",
315  "authentication_passphrase":"BadPassword",
316  "username":"Worse Password"
317  }
318  }
319 }
320 ```

324 ## SnmpV3UsmUser OOB
325 
326 Each SnmpV3UsmUser/[username] OOB is an OOB that stores the hashed passphrase(s) for one user
327 from/for the the semi-hidden /flash/app/snmpd.conf file. The snmpd_conf creates each of these OOBs
328 when a new v3 user is added, after snmpd has been run once; snmpd_conf will delete the corresponding
329 SnmpV3NewPassphrases/[username] OOB and read this user's entry from the /flash/app/snmpd.conf file
330 and store it in this OOB.
331 
332 On subsequent restarts of snmpd_conf and snmpd, snmpd_conf reads the contents of this OOB and
333 can restore them to /flash/app/snmpd.conf if not already present there; this is especially
334 helpful when backing up the system's configuration for all OOBs.
335 
336 This OOB would accessible by the UI (though there is little reason to do so),
337 but human access is restricted to Security Admin (for r/w) and Security Viewer (for read-only).
338 
339 The SnmpV3UsmUser OOB simply uses a single "GenericSettings" string field for its data, as defined in [ToolkitConfiguration.proto](../libpx/proto/ToolkitConfiguration.proto), but then adds the read/write permissions to tightly restrict access.
340 
341 The sections below describe how the SnmpV3UsmUser fields are used to support configuration
342 of the snmpd.conf file.
343 
344 
345 ### Fields for the SnmpV3UsmUser OOB
346 
347 #### generic::StringData
348 
349 The contents of this user's entry in /flash/app/snmpd.conf, given as one string of text.
350 Interested readers can dig through the Net-SNMP documentation for the breakdown of the fields here.
351 
352 #### read_permission_required
353 
354 Declares the roles necessary for users (or programs) to be able to read this OOB.
355 
356 #### write_permission_required
357 
358 Declares the more limited roles necessary for users (or programs) to be able to write to this OOB.
359 
360 #### delete_or_permission_permission_required
361 
362 Declares the roles necessary for users (or programs) to be able to delete this OOB. Note that this
363 can include the UI for advanced users.
364 
365 
366 ### Example of SnmpV3UsmUser OOB
367 
368 The following example shows a fully configured OOB, with v1 and v3 enabled,
369 two v3 users configured (one for read/write access), and both a trapsink and
370 trap session configured.
371 
372 ```json
373 {
374  "toolkitStruct":
375  {
376  "generic":
377  {
378  "StringData":"usmUser 1 3 0x80001f88804e1c3a024c0cde59 \"myRwUser\" \"myRwUser\" NULL .1.3.6.1.6.3.10.1.1.3 0x325332f1fd9e8cabe35a0bbc32c8d698ca1c1609 .1.3.6.1.6.3.10.1.2.4 0x325332f1fd9e8cabe35a0bbc32c8d698 0x"

406 ## SnmpMibs OOB
407 
408 The SnmpMibs OOB is a fixed entry OOB generated by snmpd_conf, and it provides a list of the supported
409 MIBs (both standard and Eaton-specific) for the UI. When snmpd_conf is run, it populates the table of
410 entries based on the MIB extensions that it finds in /lib/snmp; it interrogates these DLLs and from
411 that produces the list of MIB entries.
412 
413 The SnmpMibs OOB uses repeated GenericTableEntry entries,
414 as defined in [GenericTableEntry.proto](../libpx/tk_proto/hg/GenericTableEntry.proto).
415 
416 The sections below describe how the GenericTableEntry fields are used for the SnmpMibs OOB.
417 
418 
419 ### Fields for the SnmpMibs OOB
420 
421 The fields described here define one MIB file entry for the UI.
422 
423 #### label
424 
425 The name of the MIB.
426 This is a short unique label that can be used to identify the table entry.
427 
428 #### description
429 
430 A short description of the MIB file (to aid the user).
431 
432 #### value
433 
434 The CGI download script and MIB file name that can be used to access the MIB file.
435

528 ## ACCESS_CTRL/SNMP OOB
