Detailed Description

Class to support Linux Image Updates.

Class functions should be called in the following order:

Initialization Sequence: CFwLnxUtil fw_util(image_file); int rc; rc = fw_util.read_linux_image_list(); rc = check_update( log_msg ); rc = fw_util.read_sums_file(); fw_util._new_partition = fw_util.get_target_names( fw_util._current_partition ); rc = fw_util.check_update( prog_steps, log_msg );

typedef struct CFwLnxUtil::image_name_params_s CFwLnxUtil::image_name_params_t

Structure of device name data parameters.

Used to create list of new and old linux image names with partion data included. Example new/old dtb files:

new_dev = "/initlo/dtb1.img" old_dev = "/initlo/dtb0.img"

typedef std::unordered_map<std::string, image_name_params_t> CFwLnxUtil::target_name_map

Define an unordered map of target file names keyed by the image ID string.

Target file names are the actual system file names.

Example: The 'kernel' file name might be 'uImage' in the collection archive, but the update process will copy and rename this file as '/initlo/kernel0.img'or '/initlo/kernel1.img' on the flash file system.

typedef struct CFwLnxUtil::substep_params_s CFwLnxUtil::substep_params_t

Structure of image sub-step parameters.

typedef std::vector<substep_params_t> CFwLnxUtil::steps_vec_t

Define an vector of vectors that holds image names for each step of the update process.

Needed because one step of the update process (as reported to the IoT Agent) may contain one or more sub-steps to actually complete step work.

Example: The 1st step of the LinuxImage' update process has 3 sub-steps, and the following steps have one sub-step. vec[0] - 'Linux System image' substeps[0] {kernel}, subvec[1] {dtb}, subvec[2] {rootfs} vec[1] - 'EULA' substeps[0] {Eaton_EULA.txt} vec[2] - 'script' substeps[0] {post_update.sh}

Given an image name and its checksum(s), flash it to a given "target_file".

Then verify the checksum(s) of the stored file (to doublecheck). Have already checked the checksum(s) of the given image file. Sets RETVAL to -1 if the recheck fails, else leaves it at 0.

Parameters

target_fileFully qualified path to the flash file system (eg '/initlo/kernel1.img').
sha1sumSHA1 checksum expected.
sha256sumSHA256 checksum expected.
Returns
0 success, -1 on failure.

target_name_map CFwLnxUtil::_target_map

This is different from the _image_sums_map since the members here are preset expectations for the Linux update (eg, kernel, dtb, sums and signatures).
