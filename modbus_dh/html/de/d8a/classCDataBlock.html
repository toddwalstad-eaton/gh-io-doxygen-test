Detailed Description

Abstract "Data Block" class.

The idea of a "data block" applies across all protocols, although each protocol has its own idea of what actually constitutes a block of data. General properties of a data block:

  • A contiguous chunk of bytes representing the current state of that block's data, read from the device.
  • A transform of that buffer of primitive device data into an array of CDS-typed data.
  • Able to produce and handle I/O messages which [potentially] manipulate the block's data
  • Updates to the block's data may result in re-evaluation of equations, updated values being assigned to channels, etc.

See CModbusDataBlock or CIncomDataBlock for a description of the specific properties of data blocks for those protocols.

Constructor which creates the Data Block from its EDS data, gives its buffer the right size, and prepares its Data Block Transform.

The primary buffer associated with each block is a byte array representing a block of Device data. The constructor initializes the array to a size, which is normally fixed (but this is not a hard requirement; there may be blocks supported in the future whose sizes change dynamically).

uint32_t CDataBlock::priority ( px::CMonoTime  tnow,
bool  is_profiling = false 
)

Re-evaluate block priority, escalating it based on the configured priority of this Block and how much time has passed since the last update.

This recalculates the priority for this block given an updated time, using the timestamp of the last successful fetch (_last_update). The present algorithms are based on _polling_algorithm member: 1) PRIORITIZED_POLLING (default) Update blocks based on their priority and time since last update. priority = _config_priority + (tnow - _last_update) as msec * _priority_scalar But since the starting priority level, _config_priority, is equal to _priority_scalar, we can reduce the algorithm and terms to priority = (1 + tnow - _last_update) as msec * _priority_scalar 2) DETERMINISTIC_POLLING (option) Update highest priority blocks at a time interval (seconds). If _polling_algorithm member is in [1,...,60] seconds; AND this block's EDS "UpdatePriority" set to highest (i.e. 1); AND profiling has completed. Then poll immediately when the polling interval has elapsed. if ( (tnow - _last_update) > _polling_interval_msec ) priority = _Maximum_Priority; // Force poll now! else priority = 0; // Do not poll!

More items may be needed inside the data block to figure out the priority. At a minimum we'll want to use the block's base priority to figure out its starting value and scaling factor. But it may also be useful to update a block's base priority based on the channels that derive from it. I'm waving my hands on all of that for now, since it can be changed later without affecting the rest of this design, for the most part.

Parameters
tnowThe monotonic time value of "now".
is_profilingtrue if device associated with block is ungoing profiling.
Returns
The priority value for this block at this time. Will return 0 if the EDS specified a priority level of 0, which will keep this block from ever being polled. If this block hasn't been polled yet, will return a constant priority level of _Initial_Priority.

uint32_t CDataBlock::calc_priority_scalar ( uint32_t  eds_fibonacci_priority)
static

QLF to calculate the priority scalar term from the EDS' Priority term.

The EDS value is one of a Fibonacci sequence: 1, 2, 3, 5, 8, 13, ... 144 The priority scalar inverts this, so a lower EDS value results in a higher priority scalar when calculating priority escalation.

The present algorithm is _priority_scalar = (uint) (600 / EDS Priority )

For more details, see the spreadsheet and code simulation at px-toolkit/documents/prioritized_polling

Parameters
eds_fibonacci_priorityThe priority term from the EDS, where 1 is the highest priority value.
Returns
The scalar term to be used in the priority escalation calculation in the priority() function.
