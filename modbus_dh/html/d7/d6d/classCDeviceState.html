Detailed Description

Device state machine.

This uses a FSM (probably based on Boost.StateChart) to drive the state of each device independently. Remember that devices don't perform any I/O independently, rather they're reactive, and simply decide what they want to do next when asked. The FSM encapsulates all of the internal logic needed to make that decision.

I've looked at Boost.StateChart, and it would be most helpful as this FSM increases in complexity. For the simpler case, a simple switch which calls separate functions for each state will be an adequate starting point for us.

enum CDeviceState::DeviceStateEnums

The set of possible Device States we are working with.

Enumerator
DHDV_START 

The initial, transient value.

DHDV_DECLARED 

Device has been initialized but not spoken to.

DHDV_PROFILING 

We are in the process of fetching all Data Blocks once.

DHDV_POLLING 

The normal operating mode, polling for data.

DHDV_NOT_RESPONDING 

The Device has been determined to be off-line.

DHDV_BUSY 

The Device has been determined to be BUSY or temporarily unavailable.

DHDV_STATE_COUNT 

Placeholder at the end, tells us how many states there are.
